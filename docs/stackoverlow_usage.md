# Использование Stack Exchange API для обогащения данных о технологиях

## 1. Общее описание

Stack Exchange API — это публичный REST API, предоставляющий доступ к данным всех сайтов сети Stack Exchange, включая Stack Overflow. API позволяет получать информацию о вопросах, ответах, тегах, пользователях и связях между ними. Для задач нашего проекта наибольший интерес представляют данные о тегах (technologies) и их взаимосвязях, которые можно извлечь из совместного использования тегов в вопросах.

**Базовый URL API**: `https://api.stackexchange.com/2.3`

**Документация**: `https://api.stackexchange.com/docs`

API работает без обязательной аутентификации, однако без ключа приложения действует ограничение в 300 запросов в день с одного IP-адреса. При регистрации приложения на https://stackapps.com и получении ключа (параметр `key`) лимит увеличивается до 10 000 запросов в день. Для операций, требующих аутентификации пользователя, используется OAuth 2.0, но для наших целей (только чтение публичных данных) это не требуется.
## 2. Ключевые концепции API

### Пагинация

Все эндпоинты, возвращающие списки, поддерживают пагинацию через параметры `page` (номер страницы, начиная с 1) и `pagesize` (количество элементов на странице, максимум 100). В ответе присутствует поле `has_more`, указывающее, есть ли ещё страницы.

### Фильтры

API поддерживает систему фильтров для контроля того, какие поля включаются в ответ. По умолчанию используется фильтр `default`, который возвращает ограниченный набор полей. Можно создать собственный фильтр через эндпоинт `/filters/create`, указав нужные поля. Это уменьшает размер ответа и количество потребляемой квоты. Фильтр передаётся как параметр `filter` в запросе.

### Квота и троттлинг

Каждый ответ содержит поля `quota_remaining` и `quota_max`, показывающие оставшееся количество запросов. При превышении квоты API возвращает ошибку 502 с типом `throttle_violation`. Поле `backoff` в ответе указывает минимальное количество секунд, которое нужно подождать перед следующим запросом к тому же эндпоинту.

### Параметр site

Обязательный параметр `site` указывает, к какому сайту сети Stack Exchange обращается запрос. Для Stack Overflow используется значение `stackoverflow`. Другие возможные значения: `serverfault`, `superuser`, `askubuntu` и так далее.
## 3. Релевантные эндпоинты для проекта
### Получение информации о теге

**Эндпоинт**: `GET /tags/{tags}/info`

Возвращает метаданные о тегах, включая количество вопросов с этим тегом. Параметр `tags` принимает список тегов через точку с запятой (до 20 за один запрос).

**Пример запроса**:
```bash
curl https://api.stackexchange.com/2.3/tags/python/info?site=stackoverflow
```
**Пример ответа**:
```
{
    "items": [
        {
            "has_synonyms": true,
            "is_moderator_only": false,
            "is_required": false,
            "count": 2300000,
            "name": "python"
        }
    ],
    "has_more": false,
    "quota_max": 300,
    "quota_remaining": 299
}
```
Поле count показывает общее количество вопросов с данным тегом, что может служить индикатором популярности технологии.
### Получение связанных тегов

**Эндпоинт**: `GET /tags/{tags}/related`

Возвращает теги, которые чаще всего встречаются совместно с указанным тегом в вопросах. Это ключевой эндпоинт для построения связей типа USES или DEPENDS_ON_SOFTWARE в нашем графе.

### Получение синонимов тегов

**Эндпоинт**: `GET /tags/{tags}/synonyms`

Возвращает синонимы тега. Это полезно для нормализации названий технологий, поскольку одна и та же технология может упоминаться под разными тегами.

## 4. Стратегия использования для проекта
### Определение связей между технологиями

Основной сценарий — использование эндпоинта `/tags/{tag}/related` для получения списка технологий, связанных с заданной.

### Определение популярности технологии

Поле `count` из `/tags/{tag}/info` даёт абсолютную метрику популярности технологии в экосистеме разработчиков.

### Нормализация названий

Через `/tags/{tag}/synonyms` можно определить каноническое название технологии и объединить данные из разных источников. Например, `react`, `reactjs` и `react.js` — это одна технология.

## 5. Ограничения и особенности

- API отражает только экосистему разработчиков, задающих вопросы на Stack Overflow.

- Связи между тегами отражают совместное использование в вопросах, а не архитектурные зависимости. Например, теги python и pandas часто встречаются вместе, но это не означает, что Python зависит от Pandas.

- Лимит в 300 запросов без ключа быстро исчерпывается при массовом сборе данных. Рекомендуется зарегистрировать приложение и использовать ключ.

- Ответы API кэшируются на стороне сервера, поэтому повторные запросы в течение короткого времени не расходуют квоту, но и не отражают изменения в данных.
## 6. Отличия от Wikidata и DBpedia

Stack Exchange API предоставляет данные принципиально другого характера по сравнению с Wikidata и DBpedia.

Wikidata и DBpedia содержат структурированные энциклопедические знания: кто разработал технологию, на каком языке она написана, под какой лицензией распространяется, кем принадлежит. Это формальные, верифицированные факты.

Stack Exchange API отражает практическое использование технологий разработчиками. Связи между тегами показывают, какие технологии реально используются вместе в проектах, какие вызывают вопросы в связке друг с другом. Это эмпирические данные о совместном использовании.

Таким образом, Stack Exchange API дополняет Wikidata и DBpedia, добавляя слой практических связей между технологиями, который невозможно получить из энциклопедических источников. В контексте нашего проекта данные из Stack Exchange API могут использоваться для определения весов связей типа USES и USED_BY, а также для обнаружения связей, которые отсутствуют в Wikidata.
